# Spec 006 — Multi-Page

## Status

Draft

## Summary

Defines how templates expand beyond a single page when loop/table fields produce more rows than fit on one page. Only loop fields with `multiPage: true` can trigger additional pages. When rows overflow, a new page is created with the background re-rendered and the table continued (including a re-rendered header row). A hard limit (`meta.maxPages`) prevents runaway page generation. Text and image fields render only on page 1.

## Requirements

- [ ] REQ-001: Templates are single-page by default. No additional pages are created unless explicitly triggered.
- [ ] REQ-002: A loop field with `multiPage: true` triggers new page creation when its data rows exceed the available space within the bounding rectangle on the current page.
- [ ] REQ-003: On each new page, re-render the template background image before continuing table content.
- [ ] REQ-004: On each new page, re-render the table's header row at the top of the loop field's bounding rectangle before continuing with data rows.
- [ ] REQ-005: Data rows continue from where they left off on the previous page, filling the bounding rectangle top-down.
- [ ] REQ-006: Enforce `meta.maxPages` as a hard limit. If the table requires more pages than `maxPages` allows, throw an error (do NOT silently truncate).
- [ ] REQ-007: Text and image fields render only on page 1. They do not appear on subsequent pages generated by multi-page tables.

## Behaviour

### Single-Page Default

When no loop field has `multiPage: true`, the engine produces exactly one page. Rows that exceed the bounding rectangle are clipped as described in Spec 005.

### Multi-Page Trigger

1. The engine renders all page-1 content: background, text fields, image fields, and loop fields.
2. For a loop field with `multiPage: true`, after filling all available rows within the bounding rectangle on page 1, the engine checks if unrendered data rows remain.
3. If rows remain and `currentPageCount < meta.maxPages`, a new page is created.
4. If rows remain and `currentPageCount >= meta.maxPages`, the engine throws `MaxPagesExceededError`.

### New Page Construction

Each continuation page is constructed as follows:

1. **Background**: The template's `background.png` is rendered full-page, identical to page 1.
2. **Header row**: The loop field's header row is re-rendered at the top of the loop field's bounding rectangle using `headerStyle`.
3. **Data rows**: Remaining rows are rendered below the header, using `rowStyle` (with column overrides as defined in Spec 005).
4. **No other fields**: Text fields and image fields from the template are NOT rendered on continuation pages.
5. Repeat steps 1-4 if more rows remain and page limit allows.

### Page Space Calculation

On each page, the available space for data rows within the loop field bounding rectangle is:

```
availableHeight = boundingRect.height - headerRowHeight
rowsPerPage = floor(availableHeight / dataRowHeight)
```

Page 1 follows the same calculation. The header occupies space on every page.

### maxPages Enforcement

- `meta.maxPages` defaults to `1` when not specified.
- The engine counts all pages (including page 1).
- When the required page count would exceed `maxPages`, the engine throws `MaxPagesExceededError` with details including the number of pages required and the limit.
- The engine does NOT silently truncate data. This is a hard failure to ensure data integrity — the caller must be aware that not all data was rendered.

### Multiple Loop Fields

- If multiple loop fields exist on page 1, only those with `multiPage: true` can trigger additional pages.
- If more than one loop field has `multiPage: true`, they are processed sequentially. The first loop fills pages as needed, then the second loop continues on the next available page. (Each loop field occupies its own bounding rectangle, and they do not share pages for continuation.)

### Happy Path

1. Template has a loop field with 50 data rows, `multiPage: true`, and a bounding rect that fits 10 data rows per page.
2. Page 1: background + all fields (text, image, loop with header + 10 rows).
3. Page 2: background + loop header + 10 rows.
4. Page 3: background + loop header + 10 rows.
5. Page 4: background + loop header + 10 rows.
6. Page 5: background + loop header + 10 rows.
7. Total: 5 pages. If `maxPages >= 5`, success. If `maxPages < 5`, throw `MaxPagesExceededError`.

### Edge Cases

- `multiPage: false` on a loop field: overflow rows are clipped on page 1, no additional pages created.
- `maxPages: 1` with `multiPage: true` and overflow: throws `MaxPagesExceededError` (page 1 is the only allowed page, and the data requires more).
- Data fits exactly on page 1: no additional pages created regardless of `multiPage` setting.
- Empty data array with `multiPage: true`: single page with header only, no additional pages.
- Loop field bounding rect is too small for even one data row: header renders, zero data rows, remaining rows cause `MaxPagesExceededError` if `multiPage: true`.

### Error Conditions

- Required pages exceed `meta.maxPages`: throw `MaxPagesExceededError` with `{ required: number, limit: number }`.
- `maxPages` is less than 1: throw `InvalidFieldError` (at least one page must be allowed).

## Input / Output

```typescript
interface MultiPageContext {
  maxPages: number
  background: Buffer // background.png data
  fields: (TextField | ImageField | LoopField)[]
  loopData: Record<string, Record<string, string>[]> // keyed by field ID
}

interface MultiPageOutput {
  totalPages: number
  pages: PageContent[] // ordered array of page render instructions
}

interface PageContent {
  pageNumber: number
  hasBackground: boolean // always true
  fields: RenderedField[] // page 1: all fields; page 2+: loop continuation only
}

function renderMultiPage(ctx: MultiPageContext, pdfCtx: PDFContext): MultiPageOutput
```

## Acceptance Criteria

- [ ] AC-001: A template with no `multiPage: true` loop fields produces exactly one page.
- [ ] AC-002: A loop field with `multiPage: true` and more rows than fit on one page produces additional pages.
- [ ] AC-003: Each continuation page includes the re-rendered background image.
- [ ] AC-004: Each continuation page includes the loop field's header row at the top of the bounding rectangle.
- [ ] AC-005: Data rows on continuation pages pick up exactly where the previous page left off (no duplicated or skipped rows).
- [ ] AC-006: When required pages exceed `meta.maxPages`, the engine throws `MaxPagesExceededError` (not silent truncation).
- [ ] AC-007: Text fields defined in the template appear only on page 1.
- [ ] AC-008: Image fields defined in the template appear only on page 1.
- [ ] AC-009: A loop field with `multiPage: false` clips overflow rows on page 1 and does not create additional pages.
- [ ] AC-010: `maxPages: 1` with overflow data throws `MaxPagesExceededError`.
- [ ] AC-011: Data that fits exactly within page 1 produces a single page even when `multiPage: true`.

## Dependencies

- Spec 001 — .tgbl File Format (background image stored in the archive).
- Spec 002 — Template Schema (`meta.maxPages`, loop field `multiPage` flag).
- Spec 005 — Loop/Table Rendering (row rendering logic, header style, row style).

## Notes

- The decision to throw an error rather than silently truncate on `maxPages` violation is deliberate. In document generation (e.g., invoices), missing data rows would produce an incorrect document. The caller should handle the error by increasing `maxPages` or reducing data.
- Open question: should continuation pages support a different bounding rectangle for the loop field (e.g., full-page table on page 2+ but partial on page 1)? Current design reuses the same bounding rectangle on all pages.
- Open question: should the engine support a "page break" marker in the data to force a new page at a specific row? Not included in the current design.
